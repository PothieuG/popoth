# Toggle "appliqué au solde bancaire" — règles opérationnelles

Sprint **Long-Press-Toggle-Apply-To-Balance** (2026-05-23). Mécanisme : un appui long de 800 ms sur `<TransactionListItem>` toggle un timestamp `applied_to_balance_at` sur la transaction ET ajuste `bank_balances.balance` atomiquement via une composite RPC (`SECURITY DEFINER` + `FOR UPDATE` sur la row source). Permet à l'utilisateur de matérialiser explicitement, par geste tactile, les transactions qui ont effectivement impacté son solde bancaire réel.

## 1. Vue d'ensemble

- **Colonne DB** : `real_expenses.applied_to_balance_at TIMESTAMPTZ NULL` + miroir sur `real_income_entries`. NULL = non appliquée (background blanc) ; ISO string = appliquée (rouge léger expense / vert léger income).
- **Sémantique** : apply expense → `balance -= amount` ; unapply → `balance += amount`. Mirror pour income (apply → `balance += amount`).
- **2 nouvelles RPCs** (`EXPECTED_RPCS` bumped 11 → 13 dans [scripts/check-rpcs.mjs](../../scripts/check-rpcs.mjs)) :
  - `toggle_real_expense_applied_to_balance(p_expense_id uuid, p_apply boolean) RETURNS json`
  - `toggle_real_income_applied_to_balance(p_income_id uuid, p_apply boolean) RETURNS json`
  - Body : `SELECT FOR UPDATE` → guard `P0002` si déjà dans target state → `PERFORM update_bank_balance(delta, profile_id, group_id)` → `UPDATE real_*` → `RETURN json_build_object('balance', new_balance, 'applied_to_balance_at', ts|NULL)`.
- **Routes API** : `POST /api/finance/expenses/real/toggle-applied` + miroir income. Pattern `withAuthAndProfile` + `parseBody(toggleAppliedBodySchema)` + ownership check (profile_id OU group_id) + map `AppliedToggleNoOpError` → 409 `{ error: 'already-in-target-state' }`.
- **Helpers** : [lib/finance/applied-balance.ts](../../lib/finance/applied-balance.ts) (`toggleRealExpenseAppliedToBalance` + `toggleRealIncomeAppliedToBalance` + `AppliedToggleNoOpError`), re-exportés via [lib/finance/index.ts](../../lib/finance/index.ts) barrel. Le route handler appelle `ensureBankBalanceRow(filter)` ([lib/finance/bank-balance.ts](../../lib/finance/bank-balance.ts)) AVANT la RPC pour les users brand-new qui n'ont jamais set leur solde (sinon `update_bank_balance` RAISE `bank_balances row not found`).
- **Hooks** : `useRealExpenses(context).toggleApplied(id, apply)` + miroir `useRealIncomes`. Optimistic update (`onMutate` setQueryData + snapshot rollback `onError`). `onSuccess` remplace l'ISO optimiste par le timestamp serveur (`setQueryData`). `onSettled` invalide UNIQUEMENT `['bank-balance']` + `invalidateFinancialRefreshes` — **PAS** `['real-*']` (sinon refetch = skeleton qui vide la liste pendant ~300ms).
- **Sémantique `availableBalance`** : Sprint Long-Press a remplacé la formule pré-existante `bank + realIncomes - realExpenses` ([lib/finance/calc-rtl.ts::calculateAvailableCash](../../lib/finance/calc-rtl.ts)) par **`availableBalance = bank_balances.balance` pur** ([lib/finance/financial-data.ts:167](../../lib/finance/financial-data.ts)). Raison : avec le toggle user-driven, le user contrôle exactement quand une transaction "atterrit" sur le solde, donc l'affichage doit refléter la valeur literal du compte bancaire (sinon double-counting visible). `calculateAvailableCash` reste exporté pour backward-compat tests (legacy) mais n'est plus appelé en runtime.
- **Hook tactile** : [hooks/useLongPress.ts](../../hooks/useLongPress.ts) générique réutilisable (aucun pattern long-press existant pré-sprint). 800 ms delay, gère pointer events + cancel sur scroll/leave + `onContextMenu.preventDefault` post-trigger pour bloquer le menu natif iOS/Android.

## 2. ❌ À ne pas réintroduire

- ❌ **NE PAS auto-ajuster `bank_balances.balance`** lors de l'edit d'une transaction `applied_to_balance_at != NULL`. Le PUT handler ne fait aucune cascade. La friction est portée par l'UI : `<ConfirmationDialog variant="warning">` dans `TransactionTabsComponent` qui prévient « le solde ne sera pas ajusté ». L'utilisateur doit retirer du solde (long-press), modifier, puis réappliquer. **Raison** : auto-ajustement silencieux = double-débit surprise si l'user oublie le statut « appliqué ».
- ❌ **NE PAS DELETE une transaction `applied_to_balance_at != NULL`**. Les DELETE handlers ([lib/api/finance/expenses-real.ts](../../lib/api/finance/expenses-real.ts) + [income-real.ts](../../lib/api/finance/income-real.ts)) retournent 409 `{ error: 'cannot-delete-applied-transaction' }`. L'UI grise l'entrée « Supprimer » du dropdown via `disabled: isApplied`. **Raison** : sans guard, le balance resterait avec le delta de la transaction supprimée (état orphelin), invariant « toute mutation balance via composite RPC » cassé.
- ❌ **NE PAS oublier `applied_to_balance_at` dans les `.select(...)`** des handlers expenses-real / income-real / expenses-add-with-logic. Le pattern existant `.select('*, estimated_*:(name), created_by:profiles!...')` inclut automatiquement la colonne via `*` — vérifier qu'aucun `.select('id, amount, …')` explicite ne l'omet. Sinon, l'UI affichera silencieusement toutes les transactions comme non-appliquées (`isApplied = false`).
- ❌ **NE PAS retry un toggle 409** côté UI. Le 409 = no-op silencieux : la mutation concurrente a déjà mis la row dans le target state. L'optimistic update local reflète déjà la vérité, et `onSettled` invalidate refetch la query → convergence naturelle. Toast d'erreur réservé aux 4xx/5xx autres.
- ❌ **NE PAS appeler `.rpc('toggle_real_*_applied_to_balance', ...)` directement** depuis une route. Passer par `toggleRealExpenseAppliedToBalance` / `toggleRealIncomeAppliedToBalance` qui wrappent le PG `P0002` dans `AppliedToggleNoOpError` (sinon le 409 mapping côté handler est cassé).
- ❌ **NE PAS attacher le long-press au sous-bouton dropdown 3 points**. `TransactionListItem` wrappe le `<div>` actions avec `onPointerDown={(e) => e.stopPropagation()}` pour empêcher la propagation vers le root et le démarrage du timer 800ms. Sinon, hold sur les 3 points fait toggle du solde au lieu d'ouvrir le menu.
- ❌ **NE PAS dégrader `useBankBalance`** au pattern useState+useEffect (refactor TanStack Query inclus dans ce sprint). Le toggle hook invalide `['bank-balance']` pour rafraîchir EditableBalanceLine — un retour à useState casserait l'invalidation silencieusement.
- ❌ **NE PAS réintroduire `queryClient.invalidateQueries({ queryKey: ['real-expenses'/'real-incomes'] })` dans `toggleAppliedMutation.onSettled`**. Le refetch déclenché met `isFetching=true` → `TransactionTabsComponent` affiche le skeleton qui efface la liste pendant ~300ms = UX cassée pendant le geste long-press. L'optimistic update (`onMutate`) + `setQueryData(serverAppliedAt)` (`onSuccess`) convergent l'UI avec le serveur sans refetch. Le 409 (concurrent) reste safe car l'optimistic value matche déjà le target state.
- ❌ **NE PAS appeler une RPC qui écrit `bank_balances.balance` sans `ensureBankBalanceRow(filter)` préalable** quand la row peut ne pas exister. Pattern miroir `ensurePiggyBankRow` (cf. operational-rules.md §5 « Tables owner-row hybrides »). Cas vu Sprint Long-Press smoke 2026-05-23 : user fresh sans manual edit du solde → toggle → `update_bank_balance` RAISE `bank_balances row not found` → 500.
- ❌ **NE PAS rétablir `calculateAvailableCash(bank, incomes, expenses)` dans `lib/finance/financial-data.ts:167`**. La sémantique est désormais `availableBalance = userBankBalance` pur (cf. §1 « Sémantique »). Toute formule additive double-comptait avec le toggle : add transaction → realExpenses++ → solde baisse, puis apply toggle → bank baisse → solde baisse encore = -2× amount affiché. Les tests `financial-data.test.ts` (cases 1+2 golden) et `api-regressions.test.ts` (« availableBalance equals bank_balance directly ») pinnent la nouvelle sémantique.

## 4. Smoke manuel (viewport mobile 430 px DevTools)

- Long-press 800 ms sur une dépense non-appliquée → bg rouge clair + balance débité + snackbar `Dépense appliquée au solde · 50,00 €` + vibration haptique 50ms (si supporté).
- Long-press à nouveau → bg blanc + balance crédité + snackbar `Dépense retirée du solde · 50,00 €`.
- Long-press sur un revenu → bg vert clair + balance crédité.
- Scroll pendant le press → `pointercancel` natif → cancel timer + snackbar absent (pas de toggle accidentel).
- Tap rapide (< 800ms) → cancel timer, dropdown ne se déclenche pas non plus (le 3-dots a `stopPropagation`).
- Éditer une transaction appliquée → `ConfirmationDialog variant="warning"` s'ouvre avec message explicite.
- Supprimer une transaction appliquée → entrée dropdown grisée (`disabled` natif).
- Spam double long-press concurrent → second toggle 409 silencieux + invalidate query (la `FOR UPDATE` serialise serveur-side).
