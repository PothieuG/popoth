  - `toggle_carry_over_and_apply(p_expense_id, p_validate)` — bidirectionnel. `p_validate=true` : exige `is_carried_over=true AND applied_to_balance_at IS NULL`, flip vers `is_carried_over=false, applied_to_balance_at=NOW()`, débite `bank_balances.balance` par `amount`. `p_validate=false` : exige état B (`is_carried_over=false AND applied_to_balance_at NOT NULL AND carried_from_recap_id NOT NULL`), flip vers `is_carried_over=true, applied_to_balance_at=NULL`, crédite balance par `amount`. **`carried_from_recap_id` PRÉSERVÉ** dans les 2 directions — c'est la mémoire qui permet le retour arrière.   **Conventions / leçons** :

  - **Bidirectionnel via `carried_from_recap_id` comme mémoire** : la spec §5.3 demande explicitement "**Dévalider** la dépense/revenu via appui long → retour à l'état 'reporté, non compté'". Pour permettre ce retour arrière, la RPC `toggle_carry_over_and_apply(validate=false)` exige `carried_from_recap_id NOT NULL` comme guard. **Règle** : ne PAS NULL `carried_from_recap_id` à la validation (`p_validate=true`) — c'est le marqueur "cette transaction a été reportée un jour" qui rend le retour arrière possible. La colonne devient invisible UI une fois validée (le badge disparaît, le dropdown s'aligne sur les transactions normales) mais sa valeur DB persiste comme historique.

  - **Auto-détection DELETE serveur (Option A) vs param URL ou route dédiée** : choix architectural validé avec user — le serveur a déjà toute l'info (`is_carried_over` en DB) pour décider de la procédure. Le client envoie un DELETE classique, le serveur regarde et applique la bonne procédure. Avantages : (a) une seule source de vérité, (b) maintenance simplifiée (changement de règle = 1 seul endroit), (c) sécurité (client buggé ne peut pas faire la mauvaise procédure par erreur). Pattern miroir de la convention "trust internal code" CLAUDE.md.

  - **Couleur badge gris vs ambre** : ambre/jaune (prompt initial) risque de conflit visuel avec orange réservé aux budgets dans la charte couleur Popoth (CLAUDE.md §6). Gris neutre = "signal informatif, pas métier" + cohérent avec le pill gris du RecapShell header. **Règle** : pour tout futur badge "passé / info / read-only", défaut gris ; éviter ambre/orange (réservé budgets) et rouge (réservé déficit).

  - **Filter `is_carried_over=false` sur CALCULS uniquement, PAS sur LISTING** : règle critique pour ne pas casser la sémantique UX (l'utilisateur doit VOIR les carry-overs dans la liste pour pouvoir les valider/supprimer) tout en garantissant qu'ils n'impactent pas le solde/RAV (purement visuels). **Règle** : tout nouveau site qui contribue aux totaux financiers DOIT filtrer ; tout nouveau site GET qui retourne la liste au client NE doit PAS filtrer. Détection : grep `from('real_expenses')` ou `from('real_income_entries')` cross-codebase avant tout merge touchant ces tables.

  - **Modifier interdit sur carry-over (suite user feedback)** : contraintes produit explicites — "les budgets reportés au mois suivant ne peuvent pas être modifiés. Ils peuvent être que supprimer ou 'valider'". Pour modifier une carry-over, le user doit d'abord la valider (state B → état normal du mois courant), puis Modifier réapparaît. UI : spread conditionnel `...(isCurrentlyCarried ? [] : [editItem])`. API : pré-fetch is_carried_over au top des handlers PUT → 409 `cannot-edit-carried-transaction` (défense en profondeur miroir du 409 DELETE applied).

  - **Optional `onToggleCarryApplied` prop pour compat tests legacy** : faire la prop required casserait ~19 tests existants qui ne touchent pas le chemin carry-over (transaction normale, pas de `carried_from_recap_id`). Choix : optional + warn logger si manquante en prod sur une row carry-over. La prop-drill dans `TransactionTabsComponent` la fournit toujours en production. Pattern acceptable tant qu'on a un guard (warn) au runtime.

  - **Tests RPC carry-over besoin de fixture `monthly_recaps`** : le FK `carried_from_recap_id REFERENCES monthly_recaps(id)` impose qu'une row recap existe avant d'insérer une carry-over. Setup ajoute `INSERT INTO monthly_recaps(profile_id, recap_month=4, recap_year=2026, current_step='completed', completed_at=NOW())` au beforeAll + cleanup au afterAll. Pattern à mémoriser pour tout futur test gated touchant les colonnes carry-over.

  - **`pnpm db:types` regen pollue le fichier avec hint plugin + notice CLI** : la commande `pnpm supabase gen types typescript --project-id … > lib/database.types.ts` capture en stdout (a) un hint `<claude-code-hint v="1" type="plugin" value="supabase@claude-plugins-official" />` injecté par Claude Code, (b) la notice "A new version of Supabase CLI is available". Les deux corrompent le fichier TS (parse error TS1434 sur les lignes ajoutées). **Workflow** : après tout `pnpm db:types`, vérifier la fin du fichier et trimmer les lignes non-TS si présentes.
