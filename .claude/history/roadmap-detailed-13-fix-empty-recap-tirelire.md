## 11. Roadmap — Part 13/13

- ✅ **Sprint Fix-Empty-Recap-Tirelire** (livré 2026-05-19, déclenché par bug report user "0 dans mes valeurs, je ne peux pas continuer") : ferme un crash bloquant le monthly-recap Step 1 pour tout utilisateur sans ligne `piggy_bank` (nouveau compte, jamais accumulé de tirelire). **Erreur reportée user** : `Erreur lors du rééquilibrage: Erreur récupération tirelire: Cannot coerce the result to a single JSON object` au clic "Continuer" Step 1. **Root cause** : [lib/recap/step1-persist.ts:148](../../lib/recap/step1-persist.ts) `loadSnapshot` utilisait `.single()` sur `piggy_bank` qui RAISE `PGRST116` quand 0 rows existent. Le `if (piggyBankError) throw` propage l'erreur côté `handleStep1Next` (`MonthlyRecapFlow.tsx`) → alert frontend, navigation Step 2 bloquée. **Même la lecture corrigée ne suffit pas** : la RPC `update_piggy_bank_amount` ([supabase/migrations/20260506000000_create_finance_rpcs.sql:42-44](../../supabase/migrations/20260506000000_create_finance_rpcs.sql)) RAISE `'piggy_bank row not found for the given context'` quand son UPDATE affecte 0 rows. Sprint Refactor-I5 (2026-05-11) avait extrait la logique algorithme/persist mais préservé verbatim la lecture `.single()` héritée de la route god-file 740 LOC originale — le pattern n'avait jamais été stressé par un compte fresh sans piggy en local dev (les tests gated `SUPABASE_FINANCE_TESTS=1` créent toujours une ligne piggy via `resetUserFinancialState` fixture).

  **Pattern installé** : helper `ensurePiggyBankRow(filter)` idempotent (INSERT `amount=0`, ignore PG unique_violation `23505` via partial unique indexes `idx_piggy_bank_profile_id_unique` + `idx_piggy_bank_group_id_unique` cf. [supabase/migrations/20260508000000_add_piggy_bank_indexes.sql](../../supabase/migrations/20260508000000_add_piggy_bank_indexes.sql)). Appelé AVANT chaque `updatePiggyBank()` RPC dans `applyDecision`. Race-safe car la partial unique index empêche les doublons sous concurrence ; le 23505 catch est cosmétique en single-user mais correct sous double-invocation (fast retry network, devtools curl).

- ✅ **Sprint Group-Budget-Auto-Sync** (livré 2026-05-19, déclenché par bug report user "ma contribution ne bouge pas quand le groupe crée un budget") : ferme un bug où la contribution salariale d'un membre de groupe n'était jamais recalculée à la création/modification/suppression d'un item de budget de groupe (`estimated_budgets` avec `group_id IS NOT NULL`). **Root cause double** : (1) le trigger `trigger_group_budget_change` ne fire que sur UPDATE de `groups.monthly_budget_estimate`, jamais déclenché par un changement dans `estimated_budgets` ; (2) `groups.monthly_budget_estimate` était un nombre saisi manuellement au `CreateGroupForm` + éditable via `PUT /api/groups/[id]`, déconnecté de la somme réelle des items du groupe. L'intention métier utilisateur est que **le budget total du groupe = SUM(estimated_budgets WHERE group_id = X)** — la contribution doit suivre ce total automatiquement.

  **Décision architecturale** : `groups.monthly_budget_estimate` reste en DB (lu par `calculate_group_contributions` + affiché par l'UI) mais devient un **mirror auto-maintenu** par un nouveau trigger sur `estimated_budgets`. La saisie manuelle est retirée côté API (POST /api/groups + PUT /api/groups/[id]) et côté UI (CreateGroupForm). Le hook custom `useGroupContributions` (useState + AbortController, dette tech connue depuis Sprint 1.5 cf. doc2/features/group-contributions.md §11) est migré vers TanStack Query — la cascade UI passe par `invalidateFinancialRefreshes` (étendu de 3 → 4 query keys) sans plumbing manuel cross-hook.

  **Cascade DB** : INSERT/UPDATE/DELETE sur `estimated_budgets` (avec `group_id IS NOT NULL`) → `estimated_budgets_sync_group_budget` (nouveau) recompute SUM + UPDATE `groups.monthly_budget_estimate` (guard `IS DISTINCT FROM` évite no-op) → `groups_budget_contribution_recalc` (existant) PERFORM `calculate_group_contributions(group_id)` → UPSERT toutes les rows `group_contributions` du groupe. Pas de boucle (3 tables distinctes : estimated_budgets → groups → group_contributions). 1 cascade-test gated `SUPABASE_TRIGGER_TESTS=1` ajouté (Case 6 dans [lib/**tests**/trigger-behavior.test.ts](../../lib/__tests__/trigger-behavior.test.ts)) : INSERT 600€ → assert groups.monthly_budget_estimate=600 + contributions 200/400 (salaires 1000/2000) ; UPDATE 900 → 300/600 ; DELETE → 0/0.
