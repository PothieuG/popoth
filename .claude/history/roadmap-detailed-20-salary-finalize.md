## Sprints

- ✅ **Sprint Endpoints-Salary-Update-And-Finalize-V3** (sprint 08/17 Monthly Recap V3, livré 2026-05-26). 2 endpoints (`update-salaries` écran 4 + `complete` écran 5) + 2 helpers orchestrateurs (`actions-salary.ts` + `actions-finalize.ts`) + 2 nouvelles RPCs PG composites + 11 unit tests + 18 tests gated intégration. Ferme la boucle end-to-end du wizard recap : après les actions positives/négatives, le user met à jour les salaires puis "Retourne au dashboard" → snapshot appliqué aux budgets carryover + transactions validées DELETE + non-validées flaggées carried-over + recap marquée completed.

  **Architecture installée** :

  **(1) Migration `20260526000000_create_recap_finalize_rpcs.sql`** (2 RPCs, ~200 LOC SQL, SECURITY DEFINER + SET search_path = public + REVOKE PUBLIC + GRANT service_role + NOTIFY pgrst en bas) :
  - **`finalize_recap_apply_snapshot(p_recap_id uuid, p_snapshot jsonb) RETURNS json`** : itère `jsonb_each_text(p_snapshot)`, pour chaque pair (budget_id, amount) `UPDATE estimated_budgets SET carryover_spent_amount = COALESCE(.,0) + amount::numeric, carryover_applied_date = now() WHERE id = budget_id::uuid`. Collecte chaque `FOUND` dans un JSONB array. Retourne `{ applied: [{ budget_id, amount }] }`. Snapshot vide ou null → `{ applied: [] }`. Budget id inexistant → silent no-op (UPDATE matches 0 rows). Premier consumer applicatif du couple `estimated_budgets.carryover_spent_amount` + `carryover_applied_date` (les colonnes existaient dans `lib/database.types.ts` sans aucune lecture/écriture côté TS jusqu'à ce sprint).

  - **`process_recap_transactions(p_recap_id uuid, p_profile_id uuid DEFAULT NULL, p_group_id uuid DEFAULT NULL) RETURNS json`** : guard `p_profile_id XOR p_group_id` (RAISE EXCEPTION si les deux NULL ou les deux non-NULL). 4 CTEs séquentielles dans le même statement-block :
    - `WITH deleted AS (DELETE FROM real_expenses WHERE applied_to_balance_at IS NOT NULL AND is_carried_over = false AND (p_profile_id IS NULL OR profile_id = p_profile_id) AND (p_group_id IS NULL OR group_id = p_group_id) RETURNING 1) SELECT count(*) INTO v_deleted_expenses FROM deleted;`
    - Symétrique pour `real_income_entries` → `v_deleted_incomes`.
    - `WITH updated AS (UPDATE real_expenses SET is_carried_over = true, carried_from_recap_id = p_recap_id WHERE applied_to_balance_at IS NULL AND is_carried_over = false AND (...context...) RETURNING 1) SELECT count(*) INTO v_carried_expenses FROM updated;`
    - Symétrique pour `real_income_entries` → `v_carried_incomes`.

    Retourne `{ deleted_expenses, deleted_incomes, carried_expenses, carried_incomes }`. Le filtre `is_carried_over = false` est **critique** : sans lui, les rows déjà carried-over depuis un recap antérieur seraient re-flaggées (overwrite de `carried_from_recap_id`) ou DELETE'd (si l'user les avait re-validées tardivement). Test gated #10 (`prior-carried-untouched`) verrouille ce comportement.

  - **`DEFAULT NULL` sur p_profile_id + p_group_id** : choix délibéré pour que le codegen Supabase marque les 2 params `?: string` côté TS (au lieu de `string` strict). Permet d'appeler `.rpc('process_recap_transactions', { p_recap_id, p_profile_id: profile.id })` sans devoir passer `p_group_id: null` (rejeté par le type). La logique SQL utilise `IS NULL OR equality` donc l'omission TS ↔ NULL PG fonctionne. Pattern miroir de `start_monthly_recap` (sprint 05).

  - EXPECTED_RPCS **14 → 16** dans [scripts/check-rpcs.mjs](../../scripts/check-rpcs.mjs). Functions DB versionnées **21 → 23** dans pg_proc (audit `pnpm db:audit-functions`).

  **(2) Helper `lib/recap/actions-salary.ts`** (~115 LOC) :
  - **`fetchGroupMemberIds(groupId): Promise<Set<string>>`** : SELECT `id` FROM profiles WHERE group_id = X, retourne un Set. Server-side authority — jamais faire confiance au body client pour la liste des profileIds (validation gate critique pour le cas group).

  - **`executeUpdateSalaries({ context, userId, profile, recap, salaries })`** : (a) valide la cible — profile-context exige exactly `salaries.length === 1` + `salaries[0].profileId === userId`, group-context exige que chaque profileId soit dans le Set retourné par fetchGroupMemberIds (sinon `RecapActionError('invalid_target', 400, { invalid: [...] })`). (b) Pour chaque (profileId, salary) → `UPDATE profiles SET salary WHERE id` sequential. (c) Si group → `supabaseServer.rpc('calculate_group_contributions', { group_id_param })` **fail-soft** (les salaires sont déjà UPDATEd ; le trigger sur prochaine mutation de budget resync de toute façon). (d) `UPDATE monthly_recaps SET current_step = 'final_recap' WHERE id`. Throw si UPDATE final échoue (recap stuck sinon). Retourne `{ updated: salaries.length, nextStep: 'final_recap', contributionsRecalculated: boolean }`.

  **(3) Helper `lib/recap/actions-finalize.ts`** (~150 LOC) :
  - **`executeCompleteRecap({ context, profile, recap })`** : orchestrateur 3 étapes.
    - **Étape 1 (fail-soft)** : si `budget_snapshot_data` non-empty (cohésion défensive : `coerceSnapshot(raw)` accepte uniquement les Record<string, number> bien formés, sinon null), appel `finalize_recap_apply_snapshot(p_recap_id, p_snapshot)`. Erreur loggée mais swallowed — outcome.snapshotApplied=null.
    - **Étape 2 (fail-soft)** : appel `process_recap_transactions(p_recap_id, p_profile_id?, p_group_id?)`. Erreur loggée + `transactions={deleted_expenses:0, deleted_incomes:0, carried_expenses:0, carried_incomes:0}`.
    - **Étape 3 (NOT fail-soft)** : `UPDATE monthly_recaps SET completed_at = now().toISOString(), current_step = 'completed' WHERE id`. Throw si erreur — le recap reste open et l'idempotency check du prochain POST `/complete` court-circuite tout duplicate work.

    Retourne `{ recapId, completed: true, snapshotApplied, transactions }`. Type `snapshotApplied: { applied: Array<{ budget_id, amount }> } | null` discrimine empty/erreur vs success non-vide.

  - **`coerceSnapshot(raw: Json): Record<string, number> | null`** : helper privé identique au pattern de `actions-negative.ts::coerceSnapshot`. Filtre les valeurs non-`number` du JSONB (défensive — un snapshot écrit hors-spec serait silently ignoré au lieu de crash).

  **(4) Route `POST /api/monthly-recap/update-salaries/route.ts`** (~70 LOC) : `withAuthAndProfile` + `parseBody(updateSalariesBodySchema)` (schema déjà déclaré sprint 03 : `{ context, salaries: array of { profileId, salary } min(1) }`). Validation chain canonique (héritée des sprints 06/07) : context group sans group_id (400 'Pas de groupe') → `getActiveRecap` null (404 'no_active_recap') → `started_by_profile_id !== userId` (403 'not_initiator') → `current_step ≠ 'salary_update'` (409 'invalid_step' + `currentStep` echoed). Delegate à `executeUpdateSalaries` (catch RecapActionError → status code + extras dans body).

  **(5) Route `POST /api/monthly-recap/complete/route.ts`** (~90 LOC) : `withAuthAndProfile` + `parseBody(completeRecapBodySchema)` (schema sprint 03 : `{ context }` seul). Validation chain canonique + **branche idempotency** :
  - `getActiveRecap` filtre `completed_at IS NULL` (cf. sprint 06 helper). Si null, query directe `monthly_recaps` avec `.match({ profile_id|group_id, recap_month, recap_year }).not('completed_at', 'is', null).maybeSingle()`. Si match → 200 `{ data: { alreadyCompleted: true, recap: { id, completed_at, current_step } } }` (replay safe). Sinon 404 'no_active_recap'.
  - `started_by_profile_id !== userId` → 403 'not_initiator'.
  - `current_step !== 'final_recap'` → 409 'invalid_step' + `currentStep`.
  - Sinon delegate à `executeCompleteRecap` → 200 `{ data: outcome }`.

  **Conventions / leçons** :
  - **Idempotency endpoint = check DB row, pas state variable** : `complete` ne peut pas se reposer sur un compteur "a-déjà-tourné" parce que `getActiveRecap` est filtré (`completed_at IS NULL`). La disambiguation se fait par une query directe avec `.not('completed_at', 'is', null).maybeSingle()`. Pattern reproductible pour toute future API qui veut idempotency : (a) read current state, (b) si "stale" — query la "done state" explicitement, (c) si "done" → respond idempotent, (d) sinon → respond 404. Pas de side-effect dans la disambiguation (juste read).

  - **Fail-soft différencié par étape** : `apply_snapshot` et `process_transactions` sont fail-soft parce que le user a déjà vu l'écran 5 et cliqué "Retourne au dashboard" — bloquer en 500 le forcerait à re-enter et re-cliquer, friction inacceptable pour un cas-edge transient (network glitch RPC). Mais le mark-completed (`UPDATE monthly_recaps.completed_at`) DOIT throw parce que sans lui, la session next-tick re-entrerait le wizard. Trade-off documenté dans le file-header de `actions-finalize.ts`.

  - **Migration DEFAULT NULL pour codegen Supabase ergonomique** : avant l'ajout de `DEFAULT NULL` sur `p_profile_id` + `p_group_id`, le codegen marquait `process_recap_transactions: { Args: { p_group_id: string; p_profile_id: string; p_recap_id: string } }` — tous required. Côté TS, passer `null` était un type error. Solution : ajouter `DEFAULT NULL` SQL → codegen TS devient `?: string` (optional). Le runtime PG accepte `undefined` côté JS (omis du JSON payload, applique DEFAULT NULL). Pattern miroir `start_monthly_recap` (sprint 05).

  - **Régénération types depuis dev, pas prod, durant le sprint** : la migration n'est appliquée à prod qu'au sprint closeout (workflow `db push --dry-run` → confirm → `db push` → re-export baseline). Pendant le sprint, les types doivent refléter dev (sinon les calls `.rpc('finalize_recap_apply_snapshot', ...)` seraient un type error). Override via `--project-id ddehmjucyfgyppfkbddr` sur la CLI Supabase (le script `pnpm db:types` est hardcodé prod). Au sprint closeout après prod push, re-régénérer depuis prod pour aligner.

  - **`real_expenses` + `real_income_entries` ont les MÊMES 3 colonnes carryover** (`is_carried_over` bool default false, `carried_from_recap_id` uuid FK monthly_recaps, `applied_to_balance_at` timestamptz). Toute logique recap (sprint 08 + futurs) traite les 2 tables en miroir — facile d'oublier les revenus (la spec dit "dépenses ET revenus"). Test gated #1 et #7 verrouillent ce traitement symétrique.

  - **Endpoints recap totaux : 9** (start, status, transfer-surpluses-to-piggy, transform-remaining-surpluses-to-savings, refloat-from-piggy, refloat-from-savings, save-budget-snapshot, update-salaries, complete). Tous via `withAuthAndProfile` + `parseBody` Zod + même catch (`handleBadRequest` → `RecapActionError` → 500). Pattern stable cross-sprint, **9 routes en 1 répertoire**.
