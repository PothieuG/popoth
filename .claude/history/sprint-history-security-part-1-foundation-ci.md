### ✅ Fait (Sprint Polish-CI — livré 2026-05-07)

- D1 — `pnpm db:types` ne pollue plus le fichier généré quand on l'invoque normalement. Le fix déplace le `>` à l'intérieur du script ([package.json:14](package.json#L14)) : le wrapper banner pnpm va sur la stdout du terminal (visible à l'utilisateur), seul l'output du binaire `supabase gen types` atteint `lib/database.types.ts`. **Faux pas en cours de route** : tenté d'abord `pnpm --silent exec supabase gen types ...` au niveau script — ça ne marche pas car le banner est imprimé par le pnpm OUTER (celui qui spawn le script), avant même d'évaluer la valeur du script. `--silent` doit être passé par l'appelant, pas par l'appelé. **Doc à jour** : on appelle désormais `pnpm db:types` sans redirection.
- D2 — `pnpm db:check-drift` idempotent face à CRLF Windows ([scripts/check-drift.mjs:83-91](scripts/check-drift.mjs#L83)). **Root cause subtile** (pas juste byte-mismatch) : en regex JS, `.` ne matche pas `\r` (line terminator). La regex `TIMESTAMP_LINE = /^-- Generated by ... .+$/` échoue sur une ligne CRLF parce que `.+` n'avale pas le `\r` final. Conséquence : la timestamp line N'EST PAS strippée du fichier committé (CRLF), MAIS l'est du in-memory (LF) — les deux side ne s'alignent plus, offset d'une ligne, diff intégralement spurious. Fix : normaliser `\r\n → \n` AVANT `stripTimestamp`, pas après. CI Linux non affectée.
- D3 — Augmentation `lib/database.ts` (47 LOC) supprimée. **Origine** : Sprint DB / D6 avait créé l'augmentation parce que les 4 RPC C3 sont `GRANT EXECUTE TO service_role` only et n'apparaissaient pas dans `pnpm db:types`. Sprint Cleanup-Legacy / C1 a régénéré les types via `--linked` (qui les expose) → l'augmentation devient un intersection no-op (elargit `string|undefined` → `string|null|undefined`, mais aucun caller ne passe `null`). **6 imports migrés** vers `@/lib/database.types` (4 tests + 2 prod : `lib/supabase-{client,server}.ts` qui utilisaient `from './database'` relatif — la première exploration les a manqués, surfacé par le typecheck après le `git rm`). Recovery : recréer le fichier depuis le parent du commit `731cded`.
- D4 — Path filter du PR-time gate étendu ([.github/workflows/db-drift-pr.yml:14-21](.github/workflows/db-drift-pr.yml#L14)). Ajout : les 2 YAML eux-mêmes (self-monitoring — sans ça, casser le gate ne le redéclenche pas pour le détecter, cas C3 redux) + `scripts/audit-*.mjs` (B1/C2, pour cohérence avec `check-*.mjs`).

Le premier `workflow_dispatch` a surfacé 2 vrais bugs cachés depuis la création des workflows (H5/A4) — exactement le pattern C3 où la première vraie boucle de feedback CI révèle ce que la default branch couvre. **Bugs trouvés et fixés** :
  - **(a) workflow_dispatch indisponible** : le bouton "Run workflow" ne s'affiche que si la default branch GitHub a le fichier YAML. `main` n'a JAMAIS reçu les workflows (cleanup est 64 commits ahead). Fix : swap default branch `main → cleanup` via Settings → Branches. Avant ça, le cron weekly schedulé depuis H5 n'avait littéralement aucun moyen de tourner — il était silencieusement no-op depuis sa création. Action future : quand cleanup sera mergé dans main, re-swap si on veut main comme trunk.
  - **(b) `db:check-types-fresh` fail en CI** : `Cannot find project ref. Have you run supabase link?`. Root cause : `supabase gen types --linked` lit un fichier d'état local créé par `supabase link` — fonctionne sur la machine du dev (où le link a été fait une fois) mais pas dans un fresh checkout CI. Sans bloc `permissions:` explicite, le workflow hérite des read-only defaults. Fix : ajouter `permissions: { contents: read, issues: write }` au job level (pas workflow level — plus conservateur). Note : ce step n'aurait JAMAIS eu la chance de fonctionner depuis H5 — c'est le bug le plus sournois car il ne se déclenche qu'en cas de failure (et les autres checks étaient verts à chaque exec PR). Le bug (a) cachait le bug (c).
