- ✅ **Sprint Recap-Positive-Consume-Surplus — Surplus transféré disparaît vraiment de l'écran « Gestion du bilan positif »** (livré 2026-05-31, 1 commit). Bug UX rapporté par le user sur la branche `monthly_recap` : à l'étape 4 du wizard Monthly Recap V3 quand `bilanSign ∈ {'positive', 'zero'}`, le drawer « Répartir vers la tirelire » sélectionne un budget puis clique « Transférer 100€ » → rien ne se passe côté tirelire (montant inchangé), la ligne du budget reste dans la section verte « Économies après transformation », le drawer rouvert re-liste le budget déjà coché, et le bouton « Répartir un surplus vers la tirelire ? » reste actif même quand on a tout traité. Reload de la page = même état (cache navigateur exclu).

  **Cause racine — 2 problèmes superposés** :
  1. **Tracker absent** : `summary.budgets[].surplus` est calculé `max(0, estimatedAmount - spentThisMonth)` par [computeRecapSummary](../../lib/recap/calculations.ts). L'action « transférer vers la tirelire » ne touchait NI `spentThisMonth` NI un tracker côté `monthly_recaps` → chaque `/status` ou `/transfer-surpluses-to-piggy` recomputait le **même** surplus. Le client filtre `surplus > 0` → la ligne réapparaît. Le bouton se base sur `hasSurplus = surplusBudgets.length > 0` → reste à `true`.

  2. **Mauvais RPC pour l'opération** : [executeTransferSurplusesToPiggy](../../lib/recap/actions-positive.ts) utilisait `transferBudgetToPiggyBank` ([lib/finance/savings.ts](../../lib/finance/savings.ts)), qui appelle la RPC `transfer_budget_to_piggy_bank` ([supabase/migrations/20260518000000_create_savings_transfer_rpcs.sql](../../supabase/migrations/20260518000000_create_savings_transfer_rpcs.sql)). Cette RPC est sémantiquement « débite `cumulated_savings` du budget + crédite `piggy_bank.amount` ». Or le **surplus mensuel** (`estimated - spent`) est virtuel — il n'a jamais été crédité à `cumulated_savings`. Step 1 (`update_budget_cumulated_savings(-amount)`) raise « cumulated_savings would become negative » sur la quasi-totalité des scénarios réels (cumulated_savings = 0 en début de recap). Le route catch sur la boucle fail-soft → tout dans `failed[]` → `transferred[]` vide → la condition `transferred.length > 0` skip l'UPDATE du tracker → réponse HTTP 200 avec body `{ transferred: [], failed: [...] }`. Le client ne traite pas `failed[]` comme une erreur (le contrat est explicitement "fail-soft per-budget") → drawer fermé, snackbar absente, état DB inchangé.

  Les tests gated `SUPABASE_RECAP_TESTS=1` existants (3 cas dans [transfer-surpluses-to-piggy/route.integration.test.ts](../../app/api/monthly-recap/transfer-surpluses-to-piggy/__tests__/route.integration.test.ts)) ne révélaient pas le bug parce qu'ils seedaient `cumulatedSavings: 100/50/150` exactement égaux aux surplus calculés — la RPC débitait sans raise et le test passait. Scénario non-réaliste qui masquait la divergence sémantique « cumulated_savings (épargne accumulée) ≠ surplus mensuel (sous-dépensé du mois en cours) ».

  **Architecture installée** :

  **(1) Nouveau tracker JSONB côté serveur** ([supabase/migrations/20260530000000_add_piggy_transfers_data_to_monthly_recaps.sql](../../supabase/migrations/20260530000000_add_piggy_transfers_data_to_monthly_recaps.sql)). Colonne `monthly_recaps.piggy_transfers_data jsonb NOT NULL DEFAULT '{}'` qui mémorise `{ [budgetId]: amount }` des surplus déjà transférés pendant ce recap actif. Pattern miroir de `budget_snapshot_data` + `refloated_from_*` utilisé par le bilan négatif (sprint 13). Aucun backfill nécessaire (`DEFAULT '{}'` couvre les recaps existants).

  **(2) `computeRecapSummary` consomme le tracker** ([lib/recap/calculations.ts](../../lib/recap/calculations.ts)). Nouveau paramètre optionnel `piggyTransfersData?: Record<string, number>`. Dans le mapping des budgets : `effectiveSpent = spentThisMonth + (piggyTransfersData[budgetId] ?? 0)` → passe à `computeBudgetSurplus` qui clamp à 0 naturellement. Le `BudgetSummary.spentThisMonth` exposé reste la valeur brute — seul `surplus` (et `deficit` en cas d'overshoot) reflète le tracker. 3 cas test ajoutés : preserve quand `undefined`, soustrait quand `Record`, clamp quand transfert > sous-dépensé restant.

  **(3) `loadRecapSummary` forward le tracker** ([lib/recap/load-summary.ts](../../lib/recap/load-summary.ts)). Nouveau paramètre `piggyTransfersData?` sur `LoadRecapSummaryInput`. Pas d'I/O supplémentaire dans le loader — l'appelant (status route + actions-positive) qui a déjà fetch la row `monthly_recaps` passe la valeur. Le loader reste pure-load (5 reads parallèles inchangés).

  **(4) `executeTransferSurplusesToPiggy` switch vers crédit-seul + UPDATE le tracker** ([lib/recap/actions-positive.ts](../../lib/recap/actions-positive.ts)). Deux changements :
  - **Lecture** : `existingTracker = coerceSnapshot(args.recap.piggy_transfers_data) ?? {}` (réutilise `coerceSnapshot` de [deficit-math.ts](../../lib/recap/deficit-math.ts) — même coercion JSONB → `Record<string, number>` que `snapshotData`). Passe à `loadRecapSummary` pour avoir le `summaryBefore` cohérent.
  - **Action** : remplace `transferBudgetToPiggyBank` par `ensurePiggyBankRow(args.filter)` (1 fois avant le loop, idempotent via partial unique indexes) + `updatePiggyBank(args.filter, +budget.surplus)` (crédit seul du `piggy_bank.amount`, ne touche pas `cumulated_savings`). La RPC sous-jacente `update_piggy_bank_amount(p_delta, …)` existe depuis le baseline ([20260506000000_create_finance_rpcs.sql](../../supabase/migrations/20260506000000_create_finance_rpcs.sql) ligne 10).
  - **Écriture** : après le loop, merge JSON `mergedTracker = { ...existingTracker, ...transferredKV }` en JS puis UPDATE `monthly_recaps.piggy_transfers_data = mergedTracker`. Read-modify-write (pas atomic Postgres `||` merge) — acceptable car la surface est mono-initiateur, pas de cas réaliste de double-tab simultané. Persistance conditionnelle sur `transferred.length > 0` pour ne pas write quand 100 % des transferts sont en `failed[]`.

  Le second `loadRecapSummary` à la fin reçoit `mergedTracker` → le summary renvoyé au client est déjà cohérent (no extra round-trip).

  Bonus : if `ensurePiggyBankRow` raise (cas pathologique e.g. permission RLS), retour anticipé avec `failed[]` rempli pour les budgets sélectionnés + summary recalculé sur `existingTracker` — explicit failure plutôt que silent masking.

  **(5) `executeTransformRemainingToSavings` lit le tracker aussi** ([lib/recap/actions-positive.ts](../../lib/recap/actions-positive.ts)). Lit `existingTracker` du recap et le passe à `loadRecapSummary`. Le filter `b.surplus > 0` consomme naturellement les budgets déjà transférés (leur surplus est 0). Cohérence garantie : « Continuer » au bilan positif ne transforme en cumulated_savings QUE les surplus restants (pas ceux déjà en tirelire).

  **(6) `/status` route fetch le tracker + l'expose** ([app/api/monthly-recap/status/route.ts](../../app/api/monthly-recap/status/route.ts)). Le `Promise.all([loadRecapSummary, getActiveRecap])` parallèle est dropped : `loadRecapSummary` dépend maintenant du tracker, donc on séquence `getActiveRecap` d'abord (+1 RTT ~10ms acceptable). Le response `recap.piggyTransfersData` (sibling de `snapshotData`) est exposé pour parité — l'UI `BilanPositiveStep` n'en a pas besoin (elle s'appuie sur `summary.budgets[].surplus`) mais le hook le mirror dans la cache pour debug + future feature.

  **(7) `/transfer-surpluses-to-piggy` route forward + exposit** ([app/api/monthly-recap/transfer-surpluses-to-piggy/route.ts](../../app/api/monthly-recap/transfer-surpluses-to-piggy/route.ts)). Passe `recap` à l'helper, retourne `piggyTransfersData` dans `data` pour cache update fast-path. Le route `transform-remaining-surpluses-to-savings` passait déjà `recap` à son helper → 0 modif.

  **(8) Hook `useMonthlyRecap` cache update intelligent** ([hooks/useMonthlyRecap.ts](../../hooks/useMonthlyRecap.ts)). `RecapProgress` étendu avec `piggyTransfersData: Record<string, number> | null` (required, pas optional — `null` quand pas en `in_progress`). `useTransferSurplusesToPiggy.onSuccess` fait maintenant `setQueryData` avec `{ summary: data.summary, recap: { ...old.recap, piggyTransfersData: data.piggyTransfersData } }` — re-render synchrone, pas de refetch. 3 fixtures de tests RTL existantes patchées pour inclure `piggyTransfersData: null` dans leur `makeRecap()` factory (`BilanNegativeStep.test.tsx`, `FinalRecapStep.test.tsx`, `RecapWizard.test.tsx`).

  **UI inchangée** ([components/monthly-recap/steps/BilanPositiveStep.tsx](../../components/monthly-recap/steps/BilanPositiveStep.tsx)). 0 modif. Le filter `surplusBudgets = summary.budgets.filter((b) => b.surplus > 0)` + `hasSurplus = surplusBudgets.length > 0` + `disabled={isBusy || !hasSurplus}` se résolvent mécaniquement maintenant que le serveur renvoie un summary cohérent. La carte « Tirelire actuelle » (lignes 94-117) affichant `summary.piggyAmount` reflète le crédit immédiatement via le `setQueryData` du hook.

  **Patterns réutilisables installés** :
  - **JSONB tracker miroir sur `monthly_recaps`** : pour mémoriser un sous-état du recap (ce qui a été déjà transféré/snapshoté/refloué) sans toucher aux entités métier brutes (budgets, savings). Coercion via `coerceSnapshot` (originalement écrit pour `snapshotData`, désormais shared : nom géneric).
  - **Différencier sémantiquement les RPCs piggy** : `update_piggy_bank_amount(delta)` = mutation pure de la tirelire, sans contrepartie ; `transfer_budget_to_piggy_bank` = déplacement de `cumulated_savings → piggy_bank` (les deux mouvements liés). Tout flow recap qui pense "envoyer le surplus mensuel à la tirelire" doit utiliser le premier (le surplus n'est pas dans cumulated_savings).
  - **Test gated réaliste** : seed cumulated_savings=0 quand on teste le bilan positif, pas une valeur qui masque la sémantique.

  **Risque résiduel** : la migration `20260530000000_…sql` n'est pas push prod. À faire lors du push V3 batch. Sans ça, l'app prod casse au moindre /status `in_progress` (la colonne `piggy_transfers_data` absente → SELECT throws). Pas de risque tant que la branche `monthly_recap` n'est pas mergée vers `cleanup`.

  **Règles ❌ pinnées** dans [.claude/conventions/operational-rules.md](../conventions/operational-rules.md) §5 (nouvelle sous-section « Recap positive flow — surplus consumption ») :
  - **NE PAS réintroduire `transferBudgetToPiggyBank`** dans `executeTransferSurplusesToPiggy`. Cette RPC débite `cumulated_savings` qui n'a pas été crédité → raise silencieusement → tracker pas updated → UI bloquée. Utiliser `ensurePiggyBankRow + updatePiggyBank(+surplus)`.
  - **NE PAS retirer le `piggyTransfersData` argument** de `computeRecapSummary` / `loadRecapSummary`. Sans, `summary.budgets[].surplus` ne reflète pas les transferts effectués → la liste BilanPositiveStep ne se vide pas.
  - **NE PAS skip le UPDATE de `monthly_recaps.piggy_transfers_data`** dans le route handler. Le crédit `piggy_bank` est durable, mais sans le tracker, le prochain `/status` recalcule le surplus brut et la ligne réapparaît.

**Synthèse Part 27** : 1 sprint mono-feature (5 modules backend touchés + 1 migration + 6 tests modifiés/ajoutés + 0 changement UI). Pattern « JSONB tracker miroir » étendu du bilan négatif au positif. Sépare sémantiquement `update_piggy_bank_amount` (mutation pure) de `transfer_budget_to_piggy_bank` (déplacement de savings → piggy). Aucune nouvelle RPC, aucun changement contrat client.
